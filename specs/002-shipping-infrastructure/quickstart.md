# Quickstart — verifying a fresh clone deploys

How a contributor or future-self picks up this feature, provisions or verifies the Azure resource, lands the four committed artifacts, and confirms the four user stories on a real URL.

> **Note:** at the time of writing this plan, the deploy pipeline does not yet exist on this branch. This quickstart describes the path the implementation tasks (`/speckit.tasks` → `/speckit.implement`) will produce. Treat it as the contract those tasks must satisfy. Once the feature has shipped at least once, the steps below double as the "verify a fresh clone still deploys" recipe.

## Prerequisites

- **Azure CLI** (`az --version` ≥ 2.50) and `az login` on a subscription where you have Contributor or Owner on the resource group `rg-carrot-code`. For the maintainer this is the Visual Studio Enterprise subscription, currently active.
- **GitHub CLI** (`gh auth status`) with permission to manage secrets on `jasaimial/carrot-code` (for the rare manual token-rotation case).
- **Node.js ≥ 20** and `npm ≥ 10` (matches CI; needed for local build verification).
- **A modern browser** for the local-build and PWA verification steps. iOS Safari, desktop Chromium (Chrome or Edge), and desktop Firefox for the full in-scope installability matrix. Android Chrome is explicitly out of scope for v0 (see spec Non-Goals).
- **Azure portal access** for the one-time SWA-to-GitHub connection step (the only part of this feature that is NOT reproducible from committed files — see below).

## 0. The one bit that requires the Azure portal

**This step is NOT reproducible from committed files. It is a one-time UI action that produces the SWA resource and the GitHub deployment-token secret.**

Once done, every subsequent maintainer and every future re-clone of this repo gets a working deploy automatically from the committed workflow + config files — no Azure portal access is required after that, unless the resource is deleted or the token must be rotated.

Steps (Azure portal, current as of 2026-05):

1. Open <https://portal.azure.com> → **Static Web Apps** → **Create**.
2. **Subscription**: Visual Studio Enterprise (or whichever subscription is currently active on `az account show`).
3. **Resource group**: `rg-carrot-code` (create if it doesn't exist, in `westus2`).
4. **Name**: `carrot-code` (or `carrot-code-swa` if the first name is unavailable globally). Lowercase, hyphenated.
5. **Plan type**: **Free**.
6. **Region**: `westus2`. If the create dialog reports the Free SKU isn't available in `westus2`, fall back to `centralus`. Do NOT use `westus3` (not Free-supported). Do NOT switch to Standard.
7. **Deployment source**: **GitHub**. Sign in if prompted.
8. **Organization**: `jasaimial`. **Repository**: `carrot-code`. **Branch**: `001-vertical-slice` (this is the *production branch* during slice development; we'll move it to `main` after the slice merges — FR-004).
9. **Build details**: **Build Presets** → **Custom**. **App location**: `/`. **Api location**: leave empty. **Output location**: `dist`.
10. **Review + create** → **Create**.

**What Azure does behind the scenes when you click Create**:

- Provisions the SWA resource in `rg-carrot-code`.
- Generates a deployment token.
- Creates a GitHub repo secret named `AZURE_STATIC_WEB_APPS_API_TOKEN_<RANDOM>` with the token as its value. **You do not see this value.**
- Commits a workflow file at `.github/workflows/azure-static-web-apps-<random>.yml` to the configured branch (`001-vertical-slice`).
- Kicks off the first run.

**Verification immediately after Create**:

- The Azure portal → SWA → **Overview** shows the public URL.
- The GitHub repo → **Actions** tab shows the auto-generated workflow has started running.
- The GitHub repo → **Settings → Secrets and variables → Actions** shows the new `AZURE_STATIC_WEB_APPS_API_TOKEN_<RANDOM>` secret (you see the name; you do not see the value).

## 1. Review the auto-generated workflow

The Azure portal just committed `.github/workflows/azure-static-web-apps-<random>.yml` directly to the configured branch. Pull, review, and adjust if needed:

```powershell
git fetch origin
git checkout 001-vertical-slice           # or whatever branch the workflow was committed to
git pull
code .github/workflows/azure-static-web-apps-*.yml
```

Check (per [contracts/swa-deployment.md](./contracts/swa-deployment.md) and research.md Q6):

- `node-version` is `20`. Edit if Azure picked something older.
- `app_location` is `/`.
- `api_location` is `""`.
- `output_location` is `dist`.
- `skip_app_build` is absent or `false`.
- The branch trigger list matches the intended production branch.

If you edited anything, commit the change with a message naming the field(s) adjusted, and push — the workflow re-runs.

## 2. Land `staticwebapp.config.json` and the placeholder icons

Per [data-model.md](./data-model.md), commit a minimal `staticwebapp.config.json` at the repo root with `navigationFallback` + per-route headers for `/assets/*`, `/sw.js`, and `/manifest.webmanifest`.

Generate the four placeholder PNGs (any tool that emits exact pixel dimensions; for example a one-off Node + `sharp` script). Drop them at:

```text
public/icons/icon-192.png             # 192×192
public/icons/icon-512.png             # 512×512
public/icons/icon-maskable-512.png    # 512×512, full-bleed (no transparency)
public/icons/apple-touch-icon.png     # 180×180
```

Solid-color fills are sufficient (research.md Q7). The color suggested in the plan is `#2d6a3e` (matches `PALETTE_HEX.bgForest`).

Verify locally:

```powershell
npm ci
npm run build
npm run preview         # serves dist/ at http://localhost:4173
```

Open <http://localhost:4173>. Open DevTools → Application → Manifest. The four icon paths should resolve with no 404s and no missing-asset warnings. Open DevTools → Lighthouse → PWA audit. Installability should report no missing-icon errors.

Commit, push, and watch the SWA workflow re-run. The production URL now serves the same build, with placeholder icons resolving 200.

## 3. Verify US1 — maintainer can share a working URL (P1)

1. Open the production URL (the one Azure shows in the SWA → Overview).
2. Confirm the current `001-vertical-slice` build of the game loads (today: BootScene stub).
3. Push a trivial commit to the production branch (e.g., a comment-only edit to `README.md`).
4. Wait ≤ 5 minutes; reload the production URL.
5. Confirm the change is reflected.

If step 5 fails to occur within 5 minutes and the workflow run shows no failure, the SC-001 measurement is breached — investigate before declaring the feature shipped.

## 4. Verify US2 — PWA installs from a real URL (P2)

For each of the three in-scope platforms, follow the platform-native install flow against the production URL, then confirm the installed launcher opens chromeless and the game runs.

- **iOS Safari**: Share → "Add to Home Screen" → tap the new icon → confirm no Safari address bar, no bottom toolbar, custom icon (placeholder color for now).
- **Desktop Chromium (Chrome or Edge)**: URL-bar install icon or menu → "Install Carrot Code" → confirm dedicated window with custom icon.
- **Desktop Firefox**: DevTools → Application panel → confirm manifest validates (no warnings), all icon paths return 200, and the service worker reaches activated state. Firefox does not provide a Chromium-equivalent install affordance on desktop, so the test bar here is parity at the manifest/SW/asset layer per FR-010, not a literal install action.

On the desktop Chromium browser only: run Lighthouse → PWA audit against the production URL. Confirm no installability warnings, no missing-icon warnings.

Android Chrome verification is deferred to a follow-up spec when a device becomes available (spec Non-Goals).

## 5. Verify US3 — per-branch deploy previews (P3)

1. From a non-default branch (this branch itself, `002-shipping-infrastructure`, works), push a commit.
2. Wait for the workflow run to complete.
3. Find the preview URL via one of: the workflow run summary, the SWA bot comment on any open PR for the branch, or the SWA portal → Environments view.
4. Open the preview URL. Confirm it serves the branch HEAD, distinct from production.
5. Delete the branch (or close its PR). Verify the preview environment disappears from the SWA portal within minutes.

## 6. Verify US4 — service worker works offline (P4)

1. Open the production URL on a desktop Chromium browser. Wait for it to fully load (BootScene rendered).
2. Open DevTools → Application → Service Workers. Confirm a service worker is shown as "activated and is running" with a non-empty precache.
3. Open DevTools → Network → set to "Offline".
4. Reload the page. Confirm the BootScene renders within 3 seconds (SC-004) with no failed-network errors visible to the user.
5. (Optional but valuable) Repeat on a real device with airplane mode after installing the PWA per US2.

## 7. Update README and HANDOVER with the live URL

Per FR-024 / FR-025, edit `README.md` and `docs/learning/HANDOVER.md` to display the live production URL prominently. README's location should be the top of the file (e.g., a badge or a "Live demo" link); HANDOVER's location should be the TL;DR section.

## 8. (After 001 merges to `main`) Move the production branch

When `001-vertical-slice` merges to `main`:

1. In the merge PR's branch (or a follow-up PR — both acceptable), edit `.github/workflows/azure-static-web-apps-*.yml` to change every `001-vertical-slice` reference under `on.push.branches` and `on.pull_request.branches` to `main`.
2. Run `az staticwebapp update --name <swa-name> --resource-group rg-carrot-code --branch main` to update the SWA resource's production-branch setting.
3. Confirm the next push to `main` triggers a production deploy.

The Azure-provided host name does not change; the URL in `README.md` and `HANDOVER.md` remains valid.

## Troubleshooting

See [contracts/swa-deployment.md](./contracts/swa-deployment.md) for the full failure-mode taxonomy (build failure / deploy failure / auth-token failure) and the correct response for each.

For Azure-side incidents not classified above, check <https://azure.status.microsoft> before assuming a code issue.

## What this quickstart does NOT cover

- Custom domain configuration (explicit non-goal of the spec).
- Authentication / role-based access on SWA (no auth in v0; Principle XI).
- Application Insights / Functions / Cosmos DB integration (all explicit non-goals).
- Removing `netlify.toml` (separate cleanup spec if ever desired; the file stays with a "this project deploys via Azure SWA" reference comment).
- Rolling back to a previous deploy. SWA Free does not have built-in rollback; the maintainer's recourse is to `git revert` and push, which re-runs the deploy on the older HEAD. Acceptable for solo-project scale.
